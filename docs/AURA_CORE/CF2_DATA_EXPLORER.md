# 📊 CF-2 — Data Explorer (Aura Core Foundation)

> **Type:** Architecture & roadmap. **No SQL tool.** The goal is an
> **Institutional Intelligence Platform** any administrator (Chairman, Principal,
> IQAC, Admin, HOD) can use **without knowing SQL** — closer to **Power BI /
> Salesforce Reports / Dynamics Advanced Find** than a query editor.
>
> **Maps to:** Aura Insights. **Principles:** *Configuration over Hardcoding*,
> *Simplicity before Complexity*, product-agnostic, RLS-respecting, read-only.
> **Last updated:** 2026-06-26

---

## 1. The central idea — one Query Model, three front-ends

Every way of asking a question compiles into **one internal Query Model (JSON)**.
That model — not SQL — is the durable abstraction the whole capability is built on.

```
   Visual Builder   (v1 — primary)  ─┐
   Natural Language (future)        ─┼─►  Query Model (JSON)  ─►  PostgREST/Supabase compiler  ─►  Results
   Advanced SQL     (future, admin) ─┘            │
                                                  └─► Saved Views · Charts · Exports
```

- **v1 builds the Visual Builder + the Query Model + the compiler.**
- The Query Model is what gets **saved** (saved views), **charted**, **exported**, and later **generated by AI**.
- NL and SQL are **future** front-ends that produce the *same* model — so everything downstream is reused, not rebuilt.

## 2. The Query Model (JSON)

A product-agnostic, declarative description of a read query. No table names are
hardcoded in the engine — `entity`/`field` reference the **entity registry** (§4).

```jsonc
{
  "entity": "students",                 // a registered explorable entity key
  "fields": ["full_name", "roll_no", "department_id"],   // columns to return (display)
  "filters": {                          // nestable AND/OR groups
    "op": "and",
    "conditions": [
      { "field": "student_program", "operator": "eq", "value": "UG" },
      { "op": "or", "conditions": [
        { "field": "student_year", "operator": "eq", "value": 1 },
        { "field": "student_year", "operator": "eq", "value": 2 }
      ]}
    ]
  },
  "dateRange": { "field": "created_at", "from": "2026-01-01", "to": "2026-06-30" },
  "sort": [{ "field": "full_name", "dir": "asc" }],
  "groupBy": ["department_id"],         // optional
  "aggregations": [                     // optional — applied per group (or overall)
    { "fn": "count", "field": "*", "as": "students" },
    { "fn": "avg",   "field": "attendance_pct", "as": "avg_attendance" }
  ],
  "limit": 1000
}
```

**Operators (v1):** `eq, neq, gt, gte, lt, lte, like, ilike, in, is_null, not_null, between`.
**Aggregations (v1):** `count, sum, avg, min, max`.
**Filter groups:** arbitrarily nested AND / OR.

> The model is **validated** against the entity registry before execution (every
> `field` must exist and be filterable/aggregatable as claimed). Invalid models
> never reach the database.

## 3. Execution — safe by construction

The compiler turns a validated Query Model into a **supabase-js PostgREST query**
run **as the signed-in user**:

- **Read-only by nature** — only `select` is ever emitted; no DDL/DML is possible.
- **RLS-respecting & tenant-isolated** — runs under the user's policies, so a user
  only ever sees data they're already allowed to see. No cross-tenant leak path.
- **Zero SQL-injection surface** — nothing is string-built; values are bound by PostgREST.
- **Bounded** — a hard `limit` (default 1000, max configurable) and selected columns only.

**Grouping & aggregation (v1):** PostgREST doesn't do arbitrary `GROUP BY`
ergonomically, so v1 computes group-by + aggregations **in-process** over the
RLS-limited, capped result set (pure functions in the lib — unit-tested). This
keeps v1 simple and safe; heavy aggregations can later move to per-entity
database views/RPC behind the *same* Query Model (a compiler detail, invisible to
callers). *Simplicity before complexity.*

## 4. Entity Registry (product-agnostic)

Like CF-1's setting definitions, explorable data is **registered**, not hardcoded.
Campus registers its entities; Build/Field register theirs against the same engine.

`data_explorer_entities` (seed):

| Column | Meaning |
|--------|---------|
| `key` | entity id, e.g. `students` (maps to a table/view) |
| `label` | display name, e.g. "Students" |
| `category` | grouping for the picker (Academics, Finance, People…) |
| `source` | the underlying table/**view** name (views recommended for safe, shaped exposure) |
| `columns` | jsonb: `[{ key, label, type, filterable, aggregatable, groupable }]` |
| `default_date_field` | optional column used by the date-range control |
| `is_active` | toggle |

> Exposing **views** (not raw tables) is the recommended pattern — they shape and
> limit what's explorable while RLS still applies. Seeds start with a safe set
> (Students, Faculty/Staff, Admissions, Finance/Payments, Attendance, Examinations).

## 5. Features (v1)

- Entity picker (grouped by category)
- Column selection
- Filters — nested **AND / OR** groups, the operators in §2
- **Date ranges** (on the entity's date field)
- **Sorting** (multi-column)
- **Group By** + **Aggregations** (count/sum/avg/min/max)
- Results grid (paginated, capped)
- **Saved Views** + favourites (the Query Model is what's saved)
- **Export**: CSV · Excel (reuse `lib/excelXml.ts`) · PDF (print)
- **Charts: future-ready** — the grouped/aggregated result shape is chart-friendly; v1 ships the data, charts plug in later over the same result
- **Role-aware** — RLS + admin-tier gating; every run/export audited (A8 → feeds CF-4)

`data_explorer_reports` (saved views): `name`, `query_model` (jsonb), `is_favourite`,
owner + institution, RLS-scoped.

## 6. Roadmap

| Phase | Scope | Status |
|-------|-------|--------|
| **CF-2 v1** | Visual Builder + Query Model + PostgREST compiler + entity registry + saved views + export | 🔨 Building |
| CF-2.2 | Charts over the aggregated result; scheduled/emailed reports | 🔲 Future |
| CF-2.3 | **Natural Language queries** → Query Model (reuses everything downstream) | 🔲 Future |
| CF-2.4 | **Advanced SQL mode** — SUPER_ADMIN only · dedicated read-only PG role · statement timeout · row limit · DDL/DML blocked · SQL validated · every query logged/audited. Specialist platform capability, never the default. | 🔲 Future |

**Extension points designed in now:** the Query Model is front-end-agnostic (so NL
and SQL slot in as alternate producers), and the compiler is swappable (so heavy
queries can move to views/RPC without touching callers).

---

## 7. Long-term vision

Three ways to query, one engine:
1. **Visual Builder** — the primary experience (v1).
2. **Natural Language** — "students with attendance below 75% and pending fees" → Query Model.
3. **Advanced SQL** — platform administrators only, fully sandboxed (read-only role + guards + audit).

All three converge on the **Query Model**, so saved views, charts, exports, and
permissions are built once and shared. The destination is an **Institutional
Intelligence Platform** — reporting power without requiring SQL.
